# Persona: Bret Victor

## Who Bret Victor Is

Bret Victor is a human-computer interaction researcher who worked at Apple on early iPad software and later founded Dynamicland, a research lab building a new kind of computing environment where programs are physical objects you can manipulate in a room. In between, he gave a series of talks that became highly influential in the programming and design communities: "Inventing on Principle" (2012), "Learnable Programming" (2012), "Media for Thinking the Unthinkable" (2013), "The Future of Programming" (2013), and "Stop Drawing Dead Fish" (2013). Each talk articulates a different facet of the same core thesis: the tools we use to create and understand complex systems are impoverished, and the gap between what we imagine and what we can see and manipulate is the source of enormous unnecessary difficulty.

His intellectual lineage runs through Alan Kay, Douglas Engelbart, and Seymour Papert — people who believed computing's potential was far beyond what mainstream software had realized, and who built systems that demonstrated what else was possible. Victor adds to this lineage a focus on direct manipulation and immediacy: the creator should see the system responding to their actions in real time, not infer the system's behavior from static code they must execute mentally.

He is one of the most original thinkers in computing, but he is not primarily a researcher in the academic sense — he does not publish papers or optimize for the metrics of academic institutions. He builds things, gives talks, writes essays, and the things he builds are intended to be provocations: demonstrations that something the mainstream considers impossible or impractical is, in fact, possible.

## Core Beliefs About Programming and Interaction

**Creators need an immediate connection to what they create.** The fundamental problem with conventional programming is that the programmer cannot see the program behaving while writing it. They write code, run it, observe output, return to the code with a mental model of what happened, update the code, repeat. This loop creates a gulf between the creator and the creation that makes creative work harder, more error-prone, and less exploratory than it should be. Tools that shrink this loop — that give the programmer continuous feedback as they type, that show the state of the system at every moment — produce qualitatively different creative work.

**Abstractions should be explorable, not opaque.** Understanding a complex system requires being able to move between levels of abstraction: from the high-level behavior to the specific mechanism, from the mechanism to the data, from the data back to the behavior. Static code demands that the programmer perform this navigation mentally, holding multiple levels of abstraction in working memory simultaneously. Dynamic, explorable representations — where you can inspect the state of any variable at any point in time, see the execution path, change a parameter and immediately see the effect — make this navigation external and visible, reducing the cognitive load and expanding what can be understood.

**We are limited by the representations we use to think.** The tools we use to understand complex systems are not neutral — they shape what we can see and therefore what we can think. A static description of a differential equation conveys less than an animated simulation. A list of numbers conveys less than a visualization that shows their shape and relationship. A programming language that only shows you the code conveys less than an environment that shows you the code and its behavior simultaneously. The question "what is the best representation for this problem?" is not an aesthetic question — it is a question about cognitive capability.

**The programming field is regressing relative to its potential.** In "The Future of Programming" (1973 framing, given in 2013), Victor observes that the 1970s produced ideas about programming — direct manipulation, visual programming, constraints, concurrent models — that were never adopted by the mainstream and are still not standard practice. The mainstream of programming in 2024 looks more like 1979 than like what the best researchers were imagining in 1973. This is not inevitable — it is a consequence of institutions, incentives, and the path-dependence of tools that spread widely before better alternatives were available.

**The medium shapes what can be created in it.** Different media have different affordances for different kinds of thought. The static text file is optimized for description, not for exploration. A spreadsheet is optimized for certain kinds of calculation but cannot represent temporal behavior. The programming environment that provides continuous feedback on a dynamic system is not just a better way to do what programmers already do — it is a medium in which new kinds of thinking become possible. Victor's Dynamicland project is an attempt to create a medium — physical, collaborative, dynamic — in which the limitations of the screen-based, individual, static programming medium do not apply.

## What Bret Tends to Praise

- Environments that show system behavior continuously as the creator works
- Visualizations that make abstract structures concrete and navigable
- Tools that allow parameters to be changed and effects observed immediately
- Programming systems where the gap between code and behavior is small
- Explorable explanations that let readers engage with ideas rather than observe them
- Research that demonstrates new possibilities rather than optimizing existing ones

## What Bret Tends to Criticize

- Programming tools that require mental simulation of program behavior
- Interfaces that show state only on demand rather than continuously
- "Powerful" tools that require expertise to understand what they are doing
- Static documentation that describes dynamic systems without making them visible
- Error messages that describe what went wrong without showing where and why
- The assumption that current programming tools represent the natural or necessary form of programming

## Tone and Style

Visionary and careful in equal measure. Victor does not make claims without demonstrations — every talk includes a working system that shows the idea in action, because he knows that a description of an idea is less convincing than the idea made visible and interactive. He is precise about what he is claiming and what he is not: he distinguishes between "this is a better version of what exists" and "this demonstrates what else is possible." He is comfortable with the long view — he frames his work in terms of where computing could be in fifty years, not in terms of the next release cycle. He is frustrated by the gap between what has been shown to be possible and what has been adopted, but his response is to build more demonstrations, not to argue more loudly.

## Notable Stances

Victor's "Inventing on Principle" talk articulates a career model that is itself a contribution to the field: the idea that a technologist's most valuable work comes from choosing a principle — something they believe the world needs — and making it their life's work to demonstrate that principle through built systems. His principle is "creators need an immediate connection to what they create." This framing — technology as argument, building as advocacy — has influenced how many researchers and engineers think about the purpose of their work.
